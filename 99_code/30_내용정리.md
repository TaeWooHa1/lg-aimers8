## 1. FP8 vs W8A8 차이

| 비교 항목 | FP8 | W8A8 (INT8) |
|-----------|-----|-------------|
| **숫자 표현** | 부동소수점 (비균등 간격) | 정수 (균등 간격) |
| **Dynamic Range** | 넓음 ✅ | 좁음 ❌ |
| **Outlier 처리** | 잘 처리 ✅ | 취약 ❌ |
| **정확도 손실** | 적음 ✅ | 상대적으로 큼 |
| **연산 속도** | 빠름 | 매우 빠름 ✅ |
| **GPU 호환성** | H100/RTX 4090+ 필요 ❌ | T4/A100 등 광범위 ✅ |
| **캘리브레이션** | 간단/불필요 ✅ | 필수 ❌ |

- **FP8**: 8비트 부동소수점 (E4M3/E5M2). 지수부가 있어 dynamic range가 넓음
- **W8A8**: Weight=INT8 + Activation=INT8. 정수 연산으로 속도 빠름
- **T4 GPU에서는 FP8 불가, W8A8만 사용 가능**

## 2. 30_w8a8_kaggle.py 코드 실행 흐름

### 전체 흐름 (8단계)

```
패키지 설치 → 임포트/MLflow → 파라미터 설정 → GPU 확인
    → 모델 로드 (bfloat16) → 캘리브레이션 데이터 1024개 준비
    → GPTQ W8A8 양자화 (⏱️ 20~40분)
    → 모델 저장 → config.json 패치 → ZIP 생성 → MLflow 기록 → 완료
```

### 핵심 단계: GPTQ 양자화 내부 동작

```
oneshot() 호출
  ├─ 캘리브레이션 데이터 셔플 (shuffle=True)
  ├─ 각 Exaone4DecoderLayer를 순차 처리 (sequential_targets)
  │   ├─ Forward Pass로 활성화 값 수집
  │   ├─ actorder="dynamic" → 중요한 열 먼저 양자화
  │   ├─ GPTQ 알고리즘 (block_size=64) → 64열씩 양자화 + Hessian 기반 오류 보상
  │   └─ Weight → INT8, Activation → INT8 변환
  └─ embed_tokens, lm_head은 건너뜀 (ignore)
```

### config.json 패치 (2가지)
1. **max_position_embeddings**: 65536 → 32768 (KV Cache 50% 절감, 정확도 영향 없음)
2. **compressed-tensors version**: → 0.13.0 (평가 서버 호환)

## 4. 캘리브레이션 데이터 관련 Q&A

### Q: 항상 1~1024번 데이터를 고정으로 가져오는가?
- **로드 단계**: `train[:1024]` → 항상 동일한 1024개 ✅
- **셔플 단계**: `shuffle_calibration_samples=True` → 처리 **순서만** 랜덤
- 완전한 재현성이 필요하면 `shuffle=False`로 설정

### Q: 10만개 캘리브레이션은 T4x2에서 가능한가?
- **불가능** ❌
- RAM 부족 (~200GB 필요), GPTQ Hessian VRAM 초과, Kaggle 12시간 제한 초과

| 샘플 수 | 감당 가능 여부 |
|---------|-------------|
| 256 | ✅ 여유 |
| **1,024** | **✅ 현재 설정** |
| 2,048 | ⚠️ 가능하지만 빠듯 |
| 4,096 | ⚠️ OOM 위험 |
| 100,000 | ❌ 불가능 |

- **수확 체감**: 1024 → 2048은 소폭 향상, 2048 이상은 거의 차이 없음

### Q: complexity score 높은 것만 캘리브레이션하면 더 좋은가?
- **아니다** — 다양성이 난이도보다 중요
- 양자화된 모델은 모든 종류의 입력을 받으므로, 대표성 있는 분포가 핵심
- **추천**: 랜덤 샘플링 또는 stratified 샘플링

```python
# 최적: 전체에서 랜덤 샘플링
ds = load_dataset("LGAI-EXAONE/MANTA-1M", split="train")
ds = ds.shuffle(seed=42).select(range(1024))
```

### Q: 10개 모델 양자화 후 가중치 평균 (앙상블)은?
- **❌ 작동하지 않음** — 3가지 이유:
  1. **scale/zero_point가 모델마다 다름** → INT8 평균이 의미 없음
  2. **GPTQ 오류 보상 체인이 파괴됨** → 열 단위 보상이 연쇄적으로 연결되어 있음
  3. 평균 후 정확도가 오히려 **하락** 가능

## 5. Hessian 오차를 줄이는 파라미터

### GPTQModifier 파라미터

| 파라미터 | 현재값 | 역할 | 최적화 방향 |
|---------|--------|------|-----------|
| `dampening_frac` | 0.01 | H에 대각 정규화 추가 (수치 안정성) | 0.01이 최적, 변경 불필요 ✅ |
| `block_size` | 64 | 한 번에 양자화할 열 수 | 32로 줄이면 정밀도 ↑, 시간 2배 |
| `actorder` | dynamic | Hessian 대각원소 기준 양자화 순서 결정 | dynamic이 최적 ✅ |
| `sequential_targets` | 있음 | 레이어별 순차 처리 (메모리 효율) | 적용 중 ✅ |

### oneshot() 파라미터

| 파라미터 | 현재값 | 역할 | 최적화 방향 |
|---------|--------|------|-----------|
| `num_calibration_samples` | 1024 | Hessian 추정 샘플 수 (H = 1/N Σ xᵢxᵢᵀ) | 2048로 늘리면 향상, 메모리 ↑ |
| `max_seq_length` | 768 | 토큰 최대 길이 | 1024로 늘리면 향상, 메모리 ↑ |
| `shuffle_calibration_samples` | True | 캘리브레이션 순서 랜덤화 | True가 최적 ✅ |

### 더 정확하게 하고 싶다면 (시간/메모리 트레이드오프)

```python
BLOCK_SIZE = 32                    # 64 → 32 (시간 ~2배)
NUM_CALIBRATION_SAMPLES = 2048     # 1024 → 2048 (메모리 ↑)
MAX_SEQUENCE_LENGTH = 1024         # 768 → 1024 (메모리 ↑)
```

> 단, 현재 설정도 이미 가성비가 매우 좋은 설정이므로 개선 효과는 **미미** (체감 0.5~1% 이내)
