# 41_내용정리 (2025-02-25)

## 1. SmoothQuant 호환성 문제 ❌

### 원인
- EXAONE-4.0은 **QK-Reorder-Norm (Post-LN)** 구조
- SmoothQuant는 **Pre-LN** (`norm → linear`) 에서만 동작
- EXAONE은 `linear → norm` 순서 → **근본적으로 SmoothQuant 적용 불가**

### 해결
- `30_w8a8_kaggle.py`에서 SmoothQuant 관련 코드 전부 제거
- 순수 GPTQ W8A8만으로 진행

---

## 2. SmoothQuant 외 고도화 기법 검토

| 기법 | PerfNorm | SpeedNorm | 리스크 | 결론 |
|------|----------|-----------|--------|------|
| **2:4 Sparsity** | ↓ | ↑↑ | vLLM W8A8 커널 미확정 | ⚠️ 위험 |
| **ignore 확장** (layers.0, 29) | ↑ | ↓ | 낮음 | ✅ 시도 가치 |
| **KD (Knowledge Distillation)** | ↑↑↑ | = | T4x2에서 효과 미미 | ❌ 스킵 |
| **observer="mse"** | ↑ | = | 낮음 | 🤔 검토 |

## 3. 2:4 Sparsity + vLLM 호환성 조사

### 결과
- **저장 포맷** (compressed-tensors 0.13.0): ✅ 지원
- **하드웨어** (L4 GPU, SM 8.9): ✅ 지원
- **vLLM 추론 커널**: ⚠️ **W8A8 INT8 + 2:4 sparse 조합은 개발 중**
  - W4A16 + sparse: ✅ Marlin 2:4 (deprecated 예정)
  - FP8 + sparse: ✅ CUTLASS
  - **W8A8 + sparse: ⚠️ 불확실**

### 결론
- 커널 미지원 시 → 속도 향상 0 + 정확도 하락 (최악)
- 로딩 실패 시 → 제출 1회 낭비
- **권장하지 않음**

## 4. Propagating OOM 문제 (CPU RAM)

### 원인
- GPTQ의 Propagating 단계가 **CPU RAM**에 중간 활성화를 저장
- `1024 samples × 1024 tokens × 2048 hidden × 4 bytes ≈ 8.6GB` × 여러 텐서
- Kaggle RAM ~30GB 초과

### 해결 방법
| 방법 | 효과 |
|------|------|
| `NUM_CALIBRATION_SAMPLES = 512` | RAM 50% 절감 ✅ |
| `torch_dtype = float16` | T4 네이티브, 에뮬레이션 오버헤드 제거 |
| `device_map = {"": 0}` | GPU 1개만 사용 → CUDA context 해제 |

### 실제 조합별 OOM 결과
| (samples, seq_len) | 결과 |
|---------------------|------|
| (1024, 1024) | ❌ OOM |
| (512, 2048) | ❌ OOM |
| **(512, 1024)** | **✅ 성공** |
| (128, 2046) | ✅ 성공 |

## 5. bfloat16 vs float16

| 항목 | float16 | bfloat16 |
|------|---------|----------|
| 가수 (정밀도) | 10비트 (높음) | 7비트 (낮음) |
| 지수 (범위) | 5비트 (좁음) | 8비트 (넓음) |
| T4 GPU | ✅ 하드웨어 지원 | ⚠️ 소프트웨어 에뮬레이션 |
| L4 (평가서버) | ✅ | ✅ |

**결론**: T4에서 양자화할 때는 `float16`이 더 효율적

### 층화추출 vs 랜덤추출
- 512개에서는 큰 수의 법칙에 의해 **비율이 거의 동일**하게 나옴
- 층화추출의 구현 복잡성 대비 효과 거의 없음
- **랜덤 512개가 최적**

### (1024, 512) vs (512, 1024)
- **(512, 1024)이 더 좋음**: max_seq_length=512면 전체의 87%가 잘림
- 샘플 다양성보다 각 샘플의 완전성이 GPTQ Hessian에 더 중요

### 토큰 >1024 필터링
- 효과는 있으나 (truncation 방지)
- 100만 개 토크나이징에 ~1시간 소요
- **투자 대비 효과 미미 (+0.01 이하)**

## 7. 양자화 후 고도화 방법

### 불가능
- 재양자화 (INT8 → INT8 의미 없음)
- SmoothQuant 추가 (EXAONE 비호환)
- 2단계 양자화 (GPTQ 구조상 불가)

### 가능
| 방법 | 설명 |
|------|------|
| **QAT** (Quantization-Aware Training) | 가짜 양자화 노드 삽입 후 학습. llmcompressor 미지원, 복잡 |
| **양자화 전 fine-tuning** | LoRA로 학습 → 병합 → 양자화. catastrophic forgetting 주의 |
| **파라미터 변경 후 재양자화** | FP16 원본에서 설정 바꿔서 다시 양자화 (현실적 최선) |
